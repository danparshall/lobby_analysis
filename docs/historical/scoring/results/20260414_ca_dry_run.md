## What the numbers mean

PRI accessibility is the most WAF-sensitive of the three rubrics. Items Q4, Q7b–f, Q8a–o (sortable downloads), Q9 (filtered downloads), most Q13 sub-items, and Q14–Q15 evaluate the **interactive search / filter / sort UX of the live cal-access portal** — that UI is an Imperva JS challenge stub for headless fetchers (`search_01.html`, `expenditures_02.html`, identical 212-byte sha). The subagent honestly flagged these `unable_to_evaluate: true` per the locked prompt's stub-handling rule rather than guessing.

What CA *did* score on PRI accessibility (12 items) is the documentation/bulk side: data exists (Q1=1), site exists (Q2=1), historical PDFs (Q5=1), bulk download (Q6=1, Q10=1), data dictionary references (Q12=1, Q13o=1), no formal API (Q11=0), no last-updated timestamp on bulk downloads (Q16a=0). All correctly cited to the right artifacts.

PRI disclosure-law is statute-shaped, so most evidence comes from `linked_04.html` (statutes/regulations), `data_dictionary_01.html` + `linked_02.html` (CAL-ACCESS glossary), and `linked_05.html`/`linked_11.html` (FAQ). 19 unable items are likely the gift-and-conflict subsections that require the actual filing forms (Form 615, etc.) — those exist as references in `linked_03.html` (forms index) but the form bodies themselves weren't captured.

FOCAL is the most documentation-friendly rubric — only 1 unable (3.5, the "search-by-issue" UX item that requires the WAF-stubbed search UI). FOCAL scored 53/54 cleanly, including correctly recording the CA $2,000/month registration threshold in notes for indicator 1.2 (per the rubric's `scoring_guidance` instruction to record threshold values).

## Pipeline validation — what this dry-run proved

1. **Subagent dispatch works.** Three Agent subagents in parallel produced strict-schema JSON output to the expected paths.
2. **Pydantic validation catches drift.** All 174 ScoredItem rows validated cleanly on first attempt, no schema-break retries needed.
3. **Provenance stamping works.** Every row carries `model_version, prompt_sha, rubric_sha, snapshot_manifest_sha, run_id, run_timestamp, coverage_tier`. `run_metadata.json` written at the end of the pass.
4. **Stub-handling protocol is honest.** When the only relevant evidence is a WAF stub, subagents wrote `unable_to_evaluate: true` with `score: null`, exactly per the locked prompt. No guessing.
5. **Coverage-tier annotation is propagating** — every output row carries `coverage_tier=partial_waf` so downstream synthesis can distinguish "scored low because portal is weak" from "scored low because snapshot was incomplete."

## Known caveats

- **CA accessibility score is biased low** until Playwright supplements the cal-access search UI snapshot. The unable-to-evaluate rate on accessibility is partly a rubric signal (CA's UX is genuinely behind a WAF for non-browser clients) and partly a snapshot completeness gap. Distinguishing the two requires the Playwright workstream.
- **Single run**, not 3× self-consistency. Phase 3 (CA / CO / WY pilot) does the 3× temp-0 sweep. We don't yet know inter-run agreement on the 12 scored accessibility items or the 42 scored disclosure-law items.
- **No human adjudication yet.** A user review of the scored CSVs is the right next step before kicking off the pilot — if a reviewer disagrees with several scored items, that's a rubric-sharpening signal rather than per-score adjudication.
