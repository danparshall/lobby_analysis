Three framings drive the rest of the doc: it's a **data-quality pipeline, not a data-movement pipeline**; **idempotency and replay are non-negotiable**; **per-state heterogeneity is the dominant complexity**. Concrete choices include: six-stage pipeline (Discovery → Fetch → Parse → Normalize → Enrich → Publish) feeding a query layer with two surfaces (GraphQL API + MCP server); **Postgres on Neon** as the single source of truth (with `pg_trgm`, `pgvector`, full-text search — no separate graph DB, vector DB, or search engine at this scale); **Cloudflare R2** for raw artifacts (zero egress, never overwrite, version everything); FastAPI + Strawberry GraphQL + async SQLAlchemy + Pydantic; an MCP-server design with the **opposite** principles to the GraphQL API (task-shaped tools, rich self-contained returns, fuzzy NL inputs, inline provenance — 5–7 broad tools, not 20 narrow ones); Fly.io deployment; Postgres-backed work queues over Airflow/Dagster ("we don't have that complexity"); per-state rate limiting with an explicit politeness/IP-diversity escalation path. The doc is the **why**, not the **what** — implementation hasn't started.

## Why this week mattered

The compendium is the **contract** between Track A (statute → schema) and Track B (portals → data). Until it was locked, both tracks were building against a moving target. The 141-row count won't change for the foreseeable future; both tracks now have a stable surface to build against.

The v1.2 schema bump (`domain="definitions"`) was forced by a real gap: the question "is X a lobbyist under this state's law?" is structurally different from "what does a registered lobbyist have to file?" Conflating them under `domain="registration"` was producing harness-confusing row descriptions. Splitting the concept fixes the disambiguation.

## Two reframings worth flagging

1. **PRI is one of nine rubrics, not the calibration anchor.** Earlier docs anchored harness success on "match PRI" — wrong. The compendium is the universe; rubrics (PRI, FOCAL, Sunlight, Newmark, Opheim, CPI, OpenSecrets) are *projections* from a populated SMR. All nine past per-state scorings are validation signals when iterating; none privileged. Forward-looking docs (STATUS.md Current Focus, Track A description) rewritten accordingly.
2. **Multi-regime states need first-class schema support, not deferred handling.** A regime-structure survey across the 50 states identified **8 with unambiguous statutory multi-regime structures** (OH, FL, CA, NY, NJ, IL, NC, MA) plus 2–4 borderline. Of any reasonable priority 5–8 states, that's roughly half. Schema gains a `regime` field in v1.3 rather than refactoring later. Survey detail: [`docs/active/statute-extraction/results/state_regime_splitting.md`](https://github.com/danparshall/lobby_analysis/blob/statute-extraction/docs/active/statute-extraction/results/state_regime_splitting.md).
