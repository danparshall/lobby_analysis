## The five gaps

### Gap 1 — Required × Available isn't in the schema

v1.0's `FieldRequirement.status` captures the **Required** axis (`required` / `optional` / `not_applicable` / `unknown`). Nothing captures **Available**.

**Required change** — add two enum fields to `FieldRequirement`:

```python
LegalAvailability = Literal[
    "public",              # data is legally public
    "redacted",            # state law redacts it from public view
    "access_restricted",   # requires credentialed access (FOIA, etc.)
    "paper_on_request",    # technically obtainable, not online
    "unknown",
]

PracticalAvailability = Literal[
    "structured_bulk",     # CSV / JSON / API — clean extraction
    "structured_search",   # searchable database, record-by-record
    "html_search",         # HTML search UI, scrapeable
    "pdf_only",            # PDF filings, requires OCR/extraction
    "not_available",       # data unavailable despite being legally public
    "unknown",
]

class FieldRequirement(BaseModel):
    # ... existing fields ...
    legal_availability: LegalAvailability = "unknown"
    practical_availability: PracticalAvailability = "unknown"
```

**Rationale for putting both on the same model:** the N × 50 × 2 matrix output joins on (state, compendium_item) and needs both axes in a single row. Splitting them across two models (a PortalCapability companion, for example) complicates matrix generation without a clear gain. If experience shows a cleaner split is warranted, v1.2 can refactor.

**Defaults to `"unknown"`:** every existing v1.0 FieldRequirement loads cleanly; new fields fellows haven't triaged are honestly labeled.

### Gap 2 — Framework traceability hardcoded to PRI + FOCAL (clean break)

v1.0 has hardcoded framework-reference fields on three models:

- `FieldRequirement.pri_item_id: str | None`
- `FieldRequirement.focal_indicator_id: str | None`
- `RegistrationRequirement.pri_item_id: str | None`
- `ReportingPartyRequirement.pri_item_ids: list[str]`

The compendium spans nine+ frameworks. These fields don't scale.

**Required change** — introduce `FrameworkReference` and replace all four hardcoded fields with a generic `framework_references: list[FrameworkReference]`:

```python
FrameworkId = Literal[
    "pri_2010_disclosure",
    "pri_2010_accessibility",
    "focal_2024",
    "focal_2026",
    "hired_guns_2007",       # CPI 2007
    "newmark_2005",
    "newmark_2017",
    "opheim_1991",
    "sunlight_2015",
    "opensecrets_2022",
    "lda",                   # federal Lobbying Disclosure Act
    "other",                 # escape hatch for frameworks added later
]

class FrameworkReference(BaseModel):
    """A citation to an item in an external rubric/framework."""
    framework: FrameworkId
    item_id: str             # the framework's native item ID (e.g., "E1f_i", "3.3a", "A-19")
    item_text: str | None = None  # optional denormalization for debugging/export
```

Then on the three existing models:

```python
class RegistrationRequirement(BaseModel):
    # ... existing fields, minus pri_item_id ...
    framework_references: list[FrameworkReference] = Field(default_factory=list)

class ReportingPartyRequirement(BaseModel):
    # ... existing fields, minus pri_item_ids ...
    framework_references: list[FrameworkReference] = Field(default_factory=list)

class FieldRequirement(BaseModel):
    # ... existing fields, minus pri_item_id and focal_indicator_id ...
    framework_references: list[FrameworkReference] = Field(default_factory=list)
```

**Clean break rationale** (decided by Dan 2026-04-22): no live v1.0 artifacts exist requiring migration; fellows haven't populated SMRs yet. Carrying deprecated fields for backcompat adds noise without benefit.

**Breakage to existing tests:** `tests/test_models.py` currently uses `pri_item_id="E1f_i"` etc. in its FieldRequirement / RegistrationRequirement / ReportingPartyRequirement tests. These lines must be updated to use `framework_references=[FrameworkReference(framework="pri_2010_disclosure", item_id="E1f_i")]`. The v1.1 test file (`tests/test_models_v1_1.py`) includes a test that the old fields are **rejected** by the v1.1 schema — this enforces the migration.

**`FrameworkId` extensibility:** the `"other"` value handles frameworks we haven't formally listed. For anything beyond research/prior-art — e.g., a fellow discovers their state references a non-listed framework — use `framework="other"` and put the framework name in `item_text` for now; promote to a dedicated enum value in a future minor version if it gains traction.

### Gap 3 — No evidence-source tag on requirements

Some requirements can be inferred from database observation ("this field appears on every filing, so the state must require it"); others require statute reading ("this exemption is invisible in the data"). The fellows' workflow needs to distinguish these because reliability differs.

**Required change** — add evidence-source enum to `FieldRequirement` (and optionally to `RegistrationRequirement` — see Open Questions):

```python
EvidenceSource = Literal[
    "statute_verified",      # read directly from state code
    "database_inferred",     # observed in filings, law not verified
    "both",                  # statute + observation cross-confirmed
    "fellow_judgment",       # interpretive call, documented in notes
    "unknown",
]

class FieldRequirement(BaseModel):
    # ... plus the gap-1 additions ...
    evidence_source: EvidenceSource = "unknown"
    evidence_notes: str = ""
```

Default `"unknown"` for backward-compatible loading of un-triaged items.

### Gap 4 — Compendium items and matrix cells have no models

v1.0 represents requirements at the state level (via `StateMasterRecord.field_requirements`) but does not model the compendium **item** itself. The brainstorm made the compendium a first-class artifact: the N in N × 50 × 2.

**Required change** — new module `src/lobby_analysis/models/compendium.py`:

```python
"""Compendium items and matrix cells.

The compendium is the union of field-level questions across all frameworks
(PRI, FOCAL, Hired Guns, Newmark, Opheim, Sunlight, OpenSecrets, LDA).
Each compendium item is the N in the N x 50 x 2 matrix; each MatrixCell is
the projected value for a (compendium item, state) pair.
"""

from typing import Literal
from pydantic import BaseModel, Field

from lobby_analysis.models.state_master import (
    EvidenceSource,
    FieldStatus,
    FrameworkReference,
    LegalAvailability,
    PracticalAvailability,
)


CompendiumDomain = Literal[
    "registration",
    "reporting",
    "financial",
    "contact_log",
    "relationship",
    "gift",
    "revolving_door",
    "accessibility",
    "enforcement",
    "other",
]

CompendiumDataType = Literal[
    "boolean", "numeric", "categorical", "free_text", "compound"
]


class CompendiumItem(BaseModel):
    """A universal field/question in the disclosure-regime universe.

    Compendium items union across rubric frameworks. Each item may trace back
    to zero, one, or many framework items.
    """
    id: str = Field(
        description="Stable compendium-native ID (e.g., 'C_LOBBYIST_COMPENSATION')"
    )
    name: str
    description: str
    domain: CompendiumDomain
    data_type: CompendiumDataType
    framework_references: list[FrameworkReference] = Field(default_factory=list)
    maps_to_state_master_field: str | None = Field(
        default=None,
        description="Dot-path into StateMasterRecord.field_requirements[].field_path",
    )
    maps_to_filing_field: str | None = Field(
        default=None,
        description="Dot-path into LobbyingFiling (e.g., 'total_compensation')",
    )
    observable_from_database: bool = Field(
        default=False,
        description=(
            "True if this item's required-status can be inferred from "
            "observing filings; False if it requires statute reading."
        ),
    )
    notes: str = ""


class MatrixCell(BaseModel):
    """One projected row of the N x 50 x 2 matrix output.

    Denormalizes the join (StateMasterRecord, CompendiumItem) for export.
    Generated from SMR + compendium, typically by an export/projection
    function rather than stored directly.
    """
    state: str
    compendium_item_id: str
    required: FieldStatus
    legal_availability: LegalAvailability
    practical_availability: PracticalAvailability
    evidence_source: EvidenceSource
    framework_references: list[FrameworkReference] = Field(default_factory=list)
    notes: str = ""
```

**Note:** `MatrixCell` is a projection model for export. It is not stored as a primary record — it's derived from SMR + compendium on demand. Whether the projection function lives in `src/lobby_analysis/` or a separate `exports/` module is an implementation detail; keep it in the models module if it stays under 100 lines.

**Imports to update in `src/lobby_analysis/models/__init__.py`:** add `CompendiumItem`, `MatrixCell`, `CompendiumDomain`, `CompendiumDataType` to the public export list.

### Gap 5 — No per-state pipeline capability model

Dan flagged: "publish confidence/provenance alongside the data — distinguish 'state X reports 4 fields consistently' from 'state X reports 40 fields with 12% parse-error rate.'" v1.0 has per-extraction `Provenance` but no per-state aggregate capability summary.

**Required change** — new module `src/lobby_analysis/models/pipeline.py`:

```python
"""Per-state extraction pipeline capability.

Companion to StateMasterRecord: SMR captures what the state's law requires;
ExtractionCapability captures what the pipeline can actually observe.
Together they resolve matrix cells to (required, available) pairs.
"""

from datetime import datetime
from typing import Literal
from pydantic import BaseModel, Field


PortalTier = Literal[
    "clean_bulk_api",       # CSV / JSON / API — minimal-effort extraction
    "structured_search",    # searchable DB; record-by-record scrape
    "html_search",          # HTML-only search UI; scraping required
    "pdf_only",             # PDF filings; OCR + LLM extraction
    "paper_on_request",     # not online; FOIA-style request required
    "mixed",                # combination of the above across filing types
]

PipelineCadence = Literal[
    "monthly", "quarterly", "semi_annual",
    "annual", "on_change", "ad_hoc",
]


class ExtractionCapability(BaseModel):
    """What this state's extraction pipeline can and cannot do.

    One per state. Populated by the fellow owning that state. Feeds into
    the matrix export's practical_availability column.
    """
    state: str = Field(description="Two-letter abbreviation")
    portal_tier: PortalTier
    fields_extractable: list[str] = Field(
        default_factory=list,
        description="Dot-paths into LobbyingFiling/LobbyistRegistration we can reliably populate",
    )
    fields_unextractable: list[str] = Field(
        default_factory=list,
        description="Dot-paths required by state but blocked by portal",
    )
    known_limitations: list[str] = Field(
        default_factory=list,
        description="Free-text descriptions of pipeline gaps",
    )
    last_pipeline_run: datetime | None = None
    next_expected_run: datetime | None = None
    cadence: PipelineCadence | None = None
    parse_error_rate: float | None = Field(
        default=None,
        ge=0.0,
        le=1.0,
        description="Fraction of filings where extraction failed outright",
    )
    notes: str = ""
```

**Imports to update in `__init__.py`:** add `ExtractionCapability`, `PortalTier`, `PipelineCadence`.

## Open questions — surface to Dan before or during implementation

1. **Should `RegistrationRequirement` also get an `evidence_source` field?** v1.1 adds it to `FieldRequirement` only. Registration requirements are usually statute-derived (hard to DB-infer "this role IS required to register" from absence of their filings), so the field may be less useful there. Recommendation: skip for v1.1; add in v1.2 if fellows want it.

2. **Should `MatrixCell` live in `models/` or a separate `exports/` module?** It's a projection, not a stored entity. Current plan keeps it in `compendium.py` for colocation with `CompendiumItem`. Reasonable to split later.

3. **Granularity of `PortalTier`.** The six values cover the common cases. If fellows' real state coverage surfaces gaps (e.g., "SPA with no search but bulk download exists"), extend the enum in v1.2.

4. **`framework_references` plural vs. one-per-model.** Plan uses `list[FrameworkReference]` uniformly because several compendium items trace to multiple frameworks. Alternative: a dict `framework: FrameworkId -> item_ids: list[str]` for easier grouping. The list form preserves ordering (useful for "primary source" convention) and allows `item_text` per-reference. Recommendation: keep the list.

5. **Should the `"other"` framework value carry structured metadata?** Currently it relies on `item_text` to identify the framework informally. If `"other"` gets heavy use, add a `framework_other_name: str | None` field. Recommendation: wait for usage data.
