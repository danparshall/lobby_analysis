### Chunk 5 — 10-pair pre-fan-out canary

**Anti-bot incidents: 0.** Across all 10 pairs × ~5 fetches each = ~50 Justia fetches, no Cloudflare challenges fired. Operationally stable.

**Two defects surfaced on unseen states:**

#### Defect 1 — `JSONDecodeError` on prose-only model responses

AR 2010 (27.6s wall) and WV 2010 (83.5s wall) both crashed with `JSONDecodeError: Expecting value: line 1 column 1 (char 0)`. Root cause: `_parse_pass1_response` (line 369) and `_parse_response_text` (line 131) both call `json.loads(json_text)` unguarded. The regex `_FENCED_JSON_RE` looks for a ` ``` ` fenced block; when the model returns prose-only "no lobbying-disclosure regime found in this snapshot" without a fence, `json_text = text` (the bare prose) and `json.loads` raises.

AR's short wall time suggests crash in pass-1; WV's longer wall suggests crash in pass-2 or pass-3. **Proposed fix (not applied):** wrap `json.loads` calls in try/except; on `JSONDecodeError`, return `(parsed_urls=[], schema_violations=[{...truncated_response_preview}], availability=_default_availability_with_unavailable=True)`. Secondary hardening: update `api_seed_discovery_pass1_prompt.md` Rule 1 to mandate JSON-on-no-titles output. **Severity: must-fix before full fan-out** (20% crash rate on this sample → ~70 silent-fail pairs at 350-pair scale if unfixed).

#### Defect 2 — silent-empty results

WA 2010 (28.9s, 0 URLs, no pass-3) and CO 2016 (75.1s, 0 URLs, 1 pass-3 call) ran end-to-end without crashing but produced empty `parsed_urls`. WA likely: pass-1 returned empty `chosen_titles`. CO likely: pass-3 returned a `{"urls": []}` response. **Without GT for these states we cannot tell whether this is "correctly silent" (the state truly has no Justia regime for the vintage) or "silently wrong" (the model missed a real regime).** Recommend adding manual GT for ≥1 of WA / CO before full fan-out as a silent-failure detection rail.

## Trajectory of thinking

The handoff plan's narrative had two implicit predictions to validate at the 10-pair scale:

1. **The chapter-TOC ceiling generalizes**: would B4's adaptive pass-3 work on a state with 20+ section leaves under multiple chapters (OH 2010), or would the orchestrator break down under fan-out? **Validated: 30/30 on OH single-pair, 16/16 on WI 10-pair.**

2. **Anti-bot resistance holds at 10× the per-canary fetch count**: B3PW + B4 single-pair canaries did 5–8 Justia fetches each with 0 incidents; the 10-pair canary does ~50 fetches. **Validated: 0 incidents.**

What the handoff plan did NOT anticipate (or anticipated only abstractly):

3. **Unseen states without a clean Justia regime crash the parser**, not gracefully degrade. The plan's "What could change" section mentioned non-JSON responses as a vague risk; the 10-pair canary made it concrete. **This is the most important deliverable of this session** — flushing out this defect cheaply (at 10-pair scale, $0.88) before it ate $5+ at 350-pair scale.

4. **Operational fan-out wall-time is far from the ≤30s gate**. The gate was probably set in anticipation of B3 / B3PW per-pair times; B4's added pass-3 + 2-3 LLM calls per pair pushed the median wall time well above the gate. The handoff plan didn't model this — but the gate exists for a reason (a 7h sequential fan-out is operationally awkward, especially with crash-prone unseen states).

The handoff's "If you find yourself implementing something the plan doesn't anticipate, stop and surface to the user" discipline was load-bearing here. Defect 1 has an obvious cheap fix (try/except), and there was real temptation to just apply it and move on — but the plan's gate-on-user-decision policy meant I should surface it for the user to scope. The result is a cleaner handback: the user sees both defects with full context and decides scope before the next session.

## Decisions

- **Don't fix Defect 1 or Defect 2 in this session.** Surface to user with localized line numbers + proposed fixes + severity assessment in `results/20260515_b4_10pair_canary.md`.
- **Don't tighten the canary script's cost cap or concurrency settings.** Those are fan-out-planning decisions, not handoff-Chunk-5 decisions.
- **Append Chunk 4 to existing pilot-canaries file** (per handoff's explicit instruction) but start a new file for Chunk 5 (different scope: 10-pair pre-fan-out telemetry vs single-pair recall validation).
- **3 commits, not 1.** Chunk 4 is independent of Chunk 5's outcomes (Chunk 4 stood alone at $0.412 with no defects); Chunk 5 surfaces the defects that change the fan-out posture; finish-convo wraps both. Separating them keeps the git log scannable for the user.
